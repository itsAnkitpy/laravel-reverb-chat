Watch a chat message travel, end to end.

This app exists for one reason: to make real-time broadcasting stop being theory. A message goes save → event → broadcast → queue → worker → Reverb → an authorized channel → the other browser — and every step below is something we actually ran and watched move. This page is the whole journey in diagrams, with the plain-language version sitting right beside each one.

The single idea behind everything

A WebSocket is an always-open, two-way connection between a browser and the server — the “open phone line.” Everything in this project is machinery that makes that line work: something to hold it open (Reverb), something to speak into it (broadcasting an event), something to listen in each tab (Echo), and a bouncer deciding who’s allowed to tune in.

What you’ll be able to say, from memory

1 · Trace a message: save → event → broadcast → Reverb → authorized channel → other browser.

2 · Explain the private-channel auth handshake (/broadcasting/auth + channels.php).

3 · Show why one company can’t hear another’s messages — the founder’s top concern.

4 · Explain why the broadcast runs on a queue.

Why learning on Reverb still teaches you Pusher

Reverb, Pusher, and Ably are three interchangeable brands of switchboard that all speak the same protocol. We picked Reverb because it’s free, runs locally with one command, and needs no signup — but the browser code is identical if we later flip to Pusher (the tool the job posting names). Swapping is config-only.

This is also why, in a stack trace, you’ll see the class PusherBroadcaster even though we run Reverb — Reverb speaks the Pusher protocol, so Laravel authorizes through the same code path.

The split at step 3→4 is the whole point

The request finishes the moment the save is done and hands back a response (measured: 61 ms). Steps 5–8 — the slow, failable delivery — happen later, on a worker. The user never waits on the network hop to Reverb.

A separate, one-time path: the handshake

Before Agent B can hear anything, its page did a one-off POST /broadcasting/auth when it first subscribed — the bouncer approving the listener. That’s not per-message; it happens once on page load. (See the 3 network calls.)

Say it in one breath

A message is saved (a fact), broadcast as a queued job, delivered over a WebSocket to another browser, only to agents whose business owns the conversation, with the delivery done off the request, on Redis, by Horizon.

Two rules hide in that sentence, and both are interview gold:

  • Persist, then broadcast — never the reverse. If you announced first and the save failed, listeners would see a message that exists nowhere.
  • Queued, not inline. Keep the HTTP response fast; let the slow, retry-able push happen on a worker.

M1 Data model — where the wall between companies lives ✓ done

Chatway is one app used by thousands of separate companies. Company X’s agents must never see Company Y’s chats — that’s tenant isolation (a “tenant” is one company renting space in the shared app).

We enforce it the cheapest way that works: a single column, business_id, stamped on every row that must stay private. Same number = same company = allowed. This is row-level isolation — everyone shares the same tables, a column decides who sees what. For a chat widget it’s the right call.

Messages carry no business_id of their own — a message belongs to a conversation, and the conversation already carries the wall. Follow the chain: message → conversation → business_id.

Interview line

“The tenant key lives on the conversation; messages inherit isolation through their parent. I added it via a separate additive migration — never by editing a shipped one — and index()ed it because every tenant-scoped query filters on it.”

The login-as shortcut

Private channels need a logged-in user (the bouncer inspects $user->business_id). Instead of a real login screen, a dev-only route GET /login-as/{id} logs you straight in — open /login-as/1 in one window (Agent A), /login-as/2 in another (Agent B). It’s guarded with abort_unless(app()->environment('local'), 403) — a password-free login is a gaping hole anywhere but your laptop.

Save first, then announce — non-negotiable order

Announce first and the save fails → listeners see a message that exists nowhere. Save-then-announce means the announcement is always about something real.

// the one word that makes it queued, not inline
class MessageSent implements ShouldBroadcast
//                     not ShouldBroadcastNow

// the gate = the entire multi-tenant wall
Broadcast::channel('conversation.{id}', function ($user,$id){
    $c = Conversation::find($id);
    return $c && (int)$user->business_id === (int)$c->business_id;
});

Two subtleties worth their own callout

->toOthers() = “send to everyone on the channel except me.” The sender’s window already shows the message it typed, so this avoids a duplicate. It identifies “me” by the browser’s socket id.

Gotcha — toOthers() + raw fetch

Echo adds the X-Socket-Id header automatically on axios, but not on raw fetch (which M3 uses). Forget it and the sender’s window double-shows the message. Set it by hand:

headers: { 'X-Socket-Id': window.Echo.socketId() }

Interview line

“The channel-authorization callback is an authorization gate — that’s the Gates/Policies talking point — and the broadcast is queued, not inline, so the request stays fast.”

At this point the broadcast lands as a queued job. Back in M2 the queue backend was the MySQL jobs table with no worker running — so the demo showed messages 0→1 (saved instantly) and jobs 0→1 (delivery deferred). The concept made visible.

M3 Frontend & the payoff — watch it cross two windows ✓ done

The whole chat.blade.php page has exactly two jobs:

  • RECEIVE — Echo.private('conversation.1').listen('MessageSent', appendMessage)
  • SEND — fetch('/messages', {method:'POST', …}), no reload

Two browser windows each do both; Reverb relays between them. Type in Agent A → it appears in Agent B with no refresh.

The XSS defense (a free security talking point)

Incoming bodies render with textContent, never innerHTML. Send the literal text <img src=x onerror=alert(1)> and textContent shows it as harmless text; innerHTML would execute it — stored XSS. Blade {{ }} auto-escapes server-side for the same reason.

Interview line

“Stored XSS is stopped at the render boundary — textContent / {{ }}, never innerHTML with user data.”

Why this proves the gate discriminates by tenant

If both attempts had 403’d, you’d only know the endpoint denies things. Because A got 200 and C got 403 on the identical request, you’ve proven the refusal comes from who you are, nothing else.

Two facts that made a browser-free curl proof possible:

  • /broadcasting/auth is plain Laravel HTTP, not the WebSocket server. Reverb can be fully stopped and the check still fires.
  • The class running it is PusherBroadcaster even under Reverb — same Pusher code path, which is why the later Pusher swap is config-only.
Interview line — your strongest card

“A row-level business_id check in the channel callback rejected a cross-tenant subscribe with a 403. That’s the right model for a chat widget — simplest thing that works, enforced on every subscribe. I’d escalate to schema- or database-per-tenant only with a real driver: hard compliance/data-residency, noisy-neighbour performance at scale, or per-tenant backup/restore. Starting heavier without that is over-engineering.”

M5 Make it async — Redis + Horizon (the JD’s named stack) ✓ done

Sending a message does two things of very different speed. M5 splits them so the request never waits on the slow one.

  • Save the row — fast, must happen now, the user is waiting.
  • Push the broadcast to every listener — slower, can fail, can retry; no HTTP request should wait on it.

So the request does the save and returns; the push is handed to a background worker. The pipe that carries “do this later” work is a queue.

Redis = an in-memory store used as the queue backend (fast because it’s in memory). Horizon = a worker manager + dashboard for Redis queues; php artisan horizon boots workers and gives you /horizon to watch throughput and failures. It replaces plain queue:work.

Interview line

“Broadcasting is queued so the request stays fast and the slow, failable push happens on a worker. Backend is Redis; Horizon manages the pool and gives retries, backpressure, and a dashboard. That decoupling is why you never ShouldBroadcastNow on a hot path.”

Locally we used predis (pure PHP, zero setup — a composer require). Production uses the phpredis C-extension for throughput (needs PECL + compilation). Same code above it. REDIS_CLIENT=predis.

The trap — “empty Pusher console, but my message delivered?”

That’s a false positive. The sender always sees its own message (->toOthers() skips the sender; the page appends locally), and /chat renders DB history on load — neither is a real-time round-trip. Verify the actual pipe: DevTools → WS tab reads ws-ap2.pusher.com (not localhost:8080), and the Pusher Debug Console shows an API Message on private-conversation.1. We proved it by firing a real broadcast and watching Horizon publish with 0 errors.

Interview line

“The swap is config-only because Reverb already speaks the Pusher protocol — same PusherBroadcaster underneath. So I pick per project: hosted Pusher for zero-ops and pay-per-connection, self-hosted Reverb to own the infra and dodge per-message pricing, raw Socket.io only off a Laravel stack. The app code doesn’t care which carries the call.”

EVOLVED How the app improved — and the thing everyone confuses

There are two separate systems here. Keeping them apart is a senior-level clarity point — and only one of them ever changed.

A) The delivery pipe = Reverb
shoves the message down an open connection into the other browser. Here since M0, never changed. The phone line.
B) The “do it later” = queue + worker
drop a job on a queue, a worker delivers it into pipe A. The outbox + courier. This is what transitioned.

“We went from queue:work to Redis/Horizon” really means: we swapped the outbox and upgraded the courier. The pipe (Reverb) was never part of that swap.

StageOutboxCourier
First working version (M2–M4)database — MySQL jobs tablephp artisan queue:work
Now (M5)Redis — in memory (fast)php artisan horizon + /horizon dashboard

Why bother? Keep the HTTP response instant (61ms) and hand the slow, retry-able delivery to the worker. If delivery fails, the request already succeeded and the job retries on its own. That’s the JD’s exact phrase: “Redis queues, Horizon, event broadcasting.”

3 CALLS The network traffic one message makes

Open DevTools → Network, send one message, and you’ll see these — the whole real-time story in three requests.

  1. POST /messages — your send. Saves the row, queues the broadcast, returns instantly with the message as JSON.
  2. POST /broadcasting/auth — the handshake, fired once when the page first subscribes (not per message). The bouncer runs the channels.php callback and answers “yes, join conversation.1” (or 403). Look for it right after page load.
  3. The WebSocket frame — in the WS tab, click the ws://localhost:8080 connection → Messages. When the worker delivers, the MessageSent frame arrives carrying {id, body, user_id}. That frame is the message landing in the other browser.

If you can point at these three and name each, the core learning outcome is met.

WALL The tenant gate, in one function

Broadcast::channel('conversation.{id}', function ($user,$id){
    $conversation = Conversation::find($id);
    return $conversation !== null
        && (int)$user->business_id === (int)$conversation->business_id;
});
Read it in one sentence

“You may listen to this conversation only if your business is the business that owns it.” Return true → you join. Return false → 403, you never hear a byte. It runs on every subscribe attempt.

When row-level stops being enough

This is the whole multi-tenant boundary — one comparison. For a chat widget, row-level is the right default: simplest thing that works, enforced on every subscribe. The senior signal is knowing when you’d escalate:

  • Schema-per-tenant / DB-per-tenant only for hard compliance or data-residency isolation, noisy-neighbour performance at real scale, or per-tenant backup/restore needs.
  • Each buys stronger isolation at real operational cost. Starting there without a driver is over-engineering.

Home turf. Lean into it — naming the default and the escalation trigger is what separates a senior answer from a textbook one.

REVIEW What a code read-through caught (2026-07-11)

One real hole (fixed same day) and two design gaps left as lessons — exactly what a chat-company interviewer probes.

1 · Fixed — the tenant wall had an open door on the write side

Three doors — only two were guarded
📖 Read the page (/chat)
scoped to your business ✓
👂 Listen live (channel gate)
403s outsiders ✓
✍️ Write (POST /messages)
only checked the id existed — not that you own it ✗

Analogy: a receptionist checked IDs at the door for anyone coming to listen — but the mail slot let anyone drop a note inside. Agent C (business 2) could post into conversation 1 and have it broadcast to business 1’s agents.

The fix — scope the lookup to the sender’s business 
$conversation = Conversation::where('id',$id)
    ->where('business_id',$request->user()->business_id)
    ->firstOrFail(); // outsider ⇒ 404

A conversation you don’t own now behaves exactly like one that doesn’t exist — the request dies with a 404 before anything saves. (404 over 403 is a bonus: it doesn’t reveal which ids exist.)

Logged in asPOSTing intoBeforeAfter

Interview line — a found-and-fixed hole beats a build that was never wrong

“I gated the subscribe handshake first and left the write path open — my own review caught it. Fix: scope every lookup by tenant so a foreign resource 404s. The takeaway I keep: check isolation per entry point — read, write, and subscribe are three separate doors.”

2 · Documented — messages can be silently missed (the catch-up problem)

A broadcast is a loudspeaker announcement — it only reaches sockets connected at that instant. Two windows drop messages:

  • Page-load gap — /chat renders old messages, then JS connects and listens. A message sent in that half-second is too new for the render, too old for the listener — it belongs to neither.
  • Reconnect gap — the WebSocket drops in real life (lid closed, wifi blip). It reconnects, but anything sent during the outage went to a pipe that wasn’t there.
Interview line

“Broadcast is fire-and-forget, so I never treat the socket as source of truth: the DB is truth, the socket is the fast path, and the client reconciles with a catch-up fetch keyed on the last message id, deduping by id. One mechanism closes both the page-load and reconnect gaps.”

The fix — a catch-up fetch (not built)
1 · remember the last id you have
e.g. #42
↓
2 · on every (re)connect, ask HTTP
GET /messages?conversation_id=1&after=42
↓
3 · append results, resume live listening
↓
4 · dedupe by id
a message can now arrive twice (fetch + live) — skip ids you already have. This is why broadcastWith carries the id.

3 · Documented — the history query is unbounded (dies at scale)

Today: messages()->orderBy('id')->get()
fetch every message ever. 10 rows: fine. 200k rows: the page hangs.
↓fix
Initial load: newest 50
latest('id')->limit(50)->get()->reverse()
↓
“Load older”: keyset pagination
where('id','<',$oldestSeen)->latest('id')->limit(50)

Keyset (cursor) pagination = “give me 50 before this bookmark.” It jumps straight to the id via the index and stays fast at any depth. Offset (“skip 150,000 rows”) makes the DB walk past everything it skips — page 3,000 is brutal. For an ever-growing feed like chat, keyset is the right answer.

The index nugget hiding in our migration

messages.conversation_id is indexed even though we never wrote ->index() — MySQL’s InnoDB auto-indexes every foreign key. PostgreSQL does not. Port this schema to Postgres and every “messages for conversation X” query silently becomes a full-table scan until you add the index yourself.

Interview line

“Chat history is unbounded, so the page never loads it whole — newest slice first, keyset pagination backwards, never offset, because offset cost grows with depth. And I check FK columns are actually indexed per engine — MySQL auto-indexes foreign keys, Postgres doesn’t.”

ReverbLaravel’s first-party WebSocket server (the switchboard). Speaks the Pusher protocol.
EchoThe browser library that subscribes to channels and reacts to events (the receiver).
DriverWhich switchboard brand (reverb / pusher / ably); swappable, same code.
/broadcasting/authThe plain-HTTP endpoint the browser hits to ask “may I join this private channel?”; the bouncer answers 200+token or 403.
Channel callback (channels.php)The closure that authorizes/denies a subscription; here it is the tenant wall.
->toOthers()Broadcast to everyone on the channel except the sender (identified by socket id).
X-Socket-IdIdentifies the sender’s socket so toOthers() can skip it; must be set by hand with raw fetch.
CSRF tokenA per-session secret proving a request came from your own page; missing on /messages ⇒ 419.
textContent vs innerHTMLRender as text (safe) vs render as HTML (executes scripts — stored-XSS risk).
Tenant / row-level isolationMany companies share one app; a business_id column + a check keeps their data invisible to each other. The default model.
Queue / job / workerDeferred work stored on a queue and run later by a worker process.
RedisIn-memory store used here as the fast queue backend.
HorizonLaravel’s worker manager + dashboard (/horizon) for Redis queues; replaces queue:work.
predis / phpredisTwo ways PHP talks to Redis: pure-PHP package (easy) vs C-extension (fast).
MigrationVersioned code that builds/changes tables; forward-only once it has shipped.
Keyset / offset pagination“Next 50 after this id” (uses the index, fast at any depth) vs “skip N rows” (pays for every skipped row).
Catch-up fetchOn (re)connect, asking HTTP for every message after your last id, so nothing missed during a gap stays missing.
